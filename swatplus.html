Key Observation — Higher Bug Rate Than SWAT-MODFLOW

SWAT+ shows a notably higher software bug rate (12% vs 7%) than SWAT-MODFLOW, consistent with the model being a more recent and still-maturing software platform. The three-component ecosystem (QSWAT+, SWAT+ Editor, SWAT+ Toolbox) introduces more integration points where bugs can emerge between versions. The swatplus_datasets.sqlite has been a recurring source of cross-platform bugs. This is a key structural difference for the comparative paper.

§04

Issue Catalog — 9 Primary Categories

🗺️
CAT-01 · Most Frequent
Watershed Setup & HRU Delineation
ProceduralConfig
145
threads

Problems in QSWAT+ Steps 1–3: DEM delineation, stream network creation, HRU definition from land use / soil / slope rasters. Includes projection mismatches, soil coverage below 95% threshold, missing MUKEY fields, and landscape unit creation failures.

Recurring examples
  • Soil coverage <95% — QSWAT+ refuses to continue
  • MUKEY field is string instead of integer → HRU failure
  • "Create landscape" error — File C:/ path not found
  • HRU count mismatch between QSWAT+ Step 2 and SWAT+ Editor import
  • Projection of rasters not matching DEM → empty HRUs
  • STUCK IN STEP 3 — peewee.py traceback on routing unit
Systematic Solution

Pre-processing checklist: verify all rasters share the same CRS as DEM; ensure MUKEY is integer; clip soil to watershed + 5% buffer; validate coverage ≥ 95% before HRU creation. Publish a spatial data preparation SOP for QSWAT+.

CAT-02 · High Frequency
Runtime Errors & Model Crashes
BugProcedural
120
threads

The SWAT+ executable crashes during or before the simulation. Causes include malformed input files, floating overflow errors (especially in nutrient/nitrogen routines), reservoir parameter errors, and incompatible model revision vs. input file version.

Recurring examples
  • Floating overflow error — often triggered by soil data problems (NaN in N cycling)
  • Reservoir initialization parameters → runtime failure
  • HYDROGRAPH_MODULE / SD_CHANNEL error in rev. 61.0.1
  • Pathogen error: "Attempt to fetch from allocatable variable LY"
  • channel_sdmorph_day.txt does not exist (missing output file)
  • pySWATPlus: Rout_unit_ele DoesNotExist — peewee ORM failure
Systematic Solution

Implement a model-revision compatibility matrix in SWAT+ Editor. Add pre-run input validation for required tables in SQLite database. Publish known floating-overflow triggers (soil NaN, zero field capacity) with diagnostic steps in the official FAQ.

CAT-03 · Research Priority
Calibration & Sensitivity Analysis
TheoreticalProcedural
104
threads

How to calibrate SWAT+ using SWAT+ Toolbox, SWATplus-CUP, IPEAT+, or R-SWAT. Covers parameter selection, multi-site calibration, "freezing" calibrated parameters before moving downstream, and interpreting NSE/PBIAS/KGE statistics. A major research opportunity.

Recurring examples
  • Nash number varying even for fixed-parameter regions
  • How to "freeze" calibrated parameters in SWAT+ Toolbox
  • par_sample.stb not found — Toolbox sensitivity analysis
  • Decimal separator "," vs "." causes calibration to fail (locale issue)
  • Nutrient calibration after hydrology — correct sequence unclear
  • GLUE method soil_k editing with Python/code
Systematic Solution

Document a standard sequential calibration protocol: (1) water balance, (2) streamflow, (3) sediment, (4) nutrients. Compare SWAT+ Toolbox vs SWATplus-CUP vs R-SWAT performance. Publish parameter sensitivity tables specific to SWAT+ architecture (not inherited from SWAT2012).

🌾
CAT-04 · New in SWAT+
Management Schedules & Decision Tables
TheoreticalConfig
84
threads

Decision tables — the major paradigm shift in SWAT+ — are deeply misunderstood. Users struggle with condition/action syntax, irrigation trigger logic, land use change implementation, and the interaction between auto-schedules and operation schedules. No equivalent in SWAT2012.

Recurring examples
  • Decision table not working — conditions never triggered
  • Management schedule causes runtime error (SWAT+ error in mgt schedule)
  • Irrigation from two water sources with one decision table
  • Irrigation time interval option missing in SWAT+ Editor UI
  • LULCC (land use change) via decision tables — SWAT+ v2.0.4 bug
  • Testing management match — whatever condition is set, no action fires
Systematic Solution

Create a decision table cookbook with 10+ common scenarios (irrigation thresholds, harvest timing, LULCC triggers). Add a decision table debugger/tester in SWAT+ Editor that shows which conditions are being evaluated per timestep. This is a strong PhD publication topic.

CAT-05 · Data Issues
Soil, Climate & Input Data Errors
ConfigProcedural
70
threads

Errors caused by incorrect or incomplete input datasets: missing soil series in the SSURGO database, misformatted climate files, incorrect soil hydraulic properties (field capacity, saturated K), and weather generator parameter problems.

Recurring examples
  • Missing soils in SWAT+ SSURGO database — MUKEY not found
  • soils.sol problem — field capacity or saturated K = 0 → NaN crash
  • Decimal format error in climate files → flat zero precipitation output
  • SWAT+ datasets.sqlite missing default values for new WETW land use
  • usersoil table not linking to QSWAT+ project properly
  • Soil coverage 91% → must be ≥95% to proceed
Systematic Solution

Build a SWAT+ Input Data Validator: check soil property ranges, climate file format (UTF-8, decimal separator), MUKEY integer type, SSURGO coverage. Maintain an updated swatplus_datasets.sqlite with changelogs. Publish a data preparation guide with country-specific soil database tips.

CAT-06 · Scientific Gap
Output Interpretation & Water Balance
TheoreticalInterpretation
55
threads

Users misinterpret SWAT+ output files: basin_wb_aa vs basin_wb_yr averaging differences, sediment delivery ratios, in-stream water quality variables, ET computation, and SWAT+ Check requirements. The new file structure (replacing SWAT2012 output.std etc.) adds confusion.

Recurring examples
  • basin_wb_aa value ≠ average of basin_wb_yr — confused by print period
  • SWAT+ Check fails: "basin_nb_aa does not exist" — print options not set
  • Excessive sediment delivery ratio in some channels
  • In-stream nutrient routing results inconsistent after parameter changes
  • Output CSV files empty after Toolbox run — analyze output step skipped
  • ET values in output.std differ from expected basin estimates
Systematic Solution

Publish an Output Variables Guide that maps SWAT2012 output files to SWAT+ equivalents. Document print period vs averaging rules for _aa files. Add a "required print settings" checklist to SWAT+ Editor before running SWAT+ Check.

🖥️
CAT-07 · Software Issues
SWAT+ Editor / Toolbox / QSWAT+ GUI Bugs
BugProcedural
44
threads

Errors within the software interfaces themselves. The three-tool ecosystem (QSWAT+, SWAT+ Editor, SWAT+ Toolbox) creates compatibility challenges between versions. SQLite database schema changes between releases cause cascading failures.

Recurring examples
  • swatplus_datasets.sqlite missing WETW default values (v2.0.3 hotfix)
  • SWAT+ Toolbox crashes on launch / disappears on project load
  • Editor v3.0.1 still shows HYDROGRAPH_MODULE error with rev. 61.0.1
  • gwflow components not visible in QSWAT+ visualization (Step 4)
  • SWAT+ Check cannot load: table "basin_nb_aa" missing
  • Writing input files fails — routing unit table not found (peewee ORM)
Systematic Solution

Maintain a versioned compatibility matrix (QSWAT+ × Editor × Toolbox × model revision). Use semantic versioning for SQLite schema changes with migration scripts. Track GUI bugs in a public GitHub Issues tracker (already exists at github.com/swat-model).

CAT-08 · Often Overlooked
Reservoirs, Wetlands & Water Bodies
TheoreticalConfig
31
threads

Setting up reservoirs (lake shapefile vs. manual), ponds, wetlands and playas in SWAT+. The introduction of the WETW and WETM land uses in v2.0 added complexity. Users also struggle with reservoir initialization, volume-area relationships, and overflow routing.

Recurring examples
  • Reservoir initialization parameters → runtime error
  • HRUs inside lake shapefile incorrectly assigned
  • Ponds and swales — reservoir vs. pond object type confusion
  • WETW land use missing from swatplus_datasets.sqlite (v2.0 bug)
  • Reservoir volume-area relationship initialization unclear
Systematic Solution

Provide a worked example for each water body type (reservoir, pond, wetland, playa). Clarify lake shapefile → reservoir mapping rules in QSWAT+ documentation. Document WETW initialization workflow after the v2.0.3 dataset fix.

CAT-09 · Advanced
Advanced: gwflow, Nutrients, Pathogens, Sediment
TheoreticalProcedural
19
threads

Less common but scientifically significant: gwflow module integration, in-stream nutrient and pathogen processes, sediment delivery in specific channel configurations, and sub-daily simulations. Emerging area as SWAT+ matures.

Recurring examples
  • gwflow + SWAT+ Editor: wetland-GW exchange setting (on/off)
  • Pathogen error — allocatable variable LY fetch failure
  • In-stream nutrient flow/routing for channels — turn off water quality
  • Sediment production issue in QSWAT+ — parameters provided correctly
  • Sensitivity analysis of sediment yield after hydrology calibration
Systematic Solution

Develop gwflow integration tutorial for SWAT+ (standalone, not just SWAT-MODFLOW). Document the nutrient calibration sequence post-hydrology. Publish pathogen module parameter defaults and known edge cases.
