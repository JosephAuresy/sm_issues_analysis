§01

Model Context — The Original SWAT Community

Foundational Watershed Model · Land Surface Hydrology
SWAT2012 — The Reference Model

The SWAT-user group covers SWAT2012 (ArcSWAT / QSWAT) and its calibration tool SWAT-CUP, forming the oldest and largest community in the SWAT family with 5,500+ threads spanning 2009–2026. This is the reference community for the entire framework — the model from which SWAT+ was derived and to which SWAT-MODFLOW adds a groundwater component. Its issue profile is therefore the baseline for understanding the SWAT family's inherent complexity burden. The dominant characteristic is a very high procedural issue rate (55%) — the highest across all four communities studied — confirming that SWAT2012's mature documentation has successfully codified most scientific questions, leaving primarily setup and operational errors unresolved.

🔍 Key Finding — Lowest Theoretical Rate in the SWAT Family (18%)

SWAT2012 has the lowest theoretical issue rate across all four models (18%), reflecting 30+ years of documentation, publications, and community knowledge. The theory manual, I/O documentation, and SWAT-CUP guides have successfully converted most theoretical questions into procedural ones. This confirms the framework's core proposition in reverse: mature documentation lowers the theoretical frontier, pushing complexity into the procedural domain. The very high procedural rate (55%) means users primarily struggle with how to do things correctly, not what things mean physically — a direct consequence of SWAT's maturity. This contrasts sharply with MODFLOW (43% theoretical), which after 40 years still demands hydrogeological conceptualisation that documentation cannot replace.

Issue Catalog — 8 Primary Categories

🗺️
CAT-01 · Most Frequent
Watershed Setup & HRU Delineation
ProceduralConfig
1,045
threads

DEM delineation, stream network creation, HRU definition (soil × land use × slope), ArcSWAT/QSWAT interface setup, and soil/land use reclassification table errors. The dominant issue across the community's entire history, driven by new users from diverse GIS backgrounds.

Recurring examples
  • ArcSWAT not installing with ArcGIS 10.8 — compatibility matrix
  • "SWAT2012.mdb database does not exist" — project path broken
  • HRU threshold: too many or too few HRUs created
  • Landuse raster returning NaN when read by ArcSWAT
  • QSWAT "User Soil" table error during file upload
  • No point sources or inlets error — topology issue in stream network
  • Adding point source reduces outlet flow — FLODAY sign convention
Systematic Solution

ArcGIS × ArcSWAT × SWAT revision compatibility matrix. Pre-setup checklist: DEM resolution, projection consistency, soil/landuse coverage ≥ 95%. Flowchart for HRU threshold selection. Point source import SOP with FLODAY sign convention explained.

🌧️
CAT-02 · Second Most Frequent
Weather Data & Climate Input Errors
ConfigProcedural
880
threads

The most format-sensitive category. SWAT's weather input files (pcp, tmp, slr, hmd, wnd) and weather generator (WGEN) have strict column-width and decimal-format requirements. A single malformed date or missing -99 fill value crashes the entire run. Affects all users globally.

Recurring examples
  • "UN UNKNOWN ERROR OCCURRED" — WGEN table import failure (keyboard locale)
  • slr.slr format error — missing days should be -99.000 not blank
  • Multi-weather station: LAT/LONG must be decimal degrees, not projected CRS
  • ArcSWAT2012 vs 2009: pcplocation.txt format changed (dbf → txt)
  • Weather data format: "The table does not have correct fields" — OID DATE PCP
  • SWAT only using one weather station despite multiple defined
Systematic Solution

Publish a validated weather file format template for each file type with an automated validator (Python/R) that checks column widths, fill values, date continuity, and coordinate format. This single tool would eliminate ~15% of all SWAT-user threads.

💥
CAT-03 · High Frequency
Runtime Errors & Model Crashes
BugProcedural
740
threads

SWAT executable crashes with Fortran runtime errors. The most common triggers are soil data formatting errors (causing NaN in nitrogen cycling), management operation timing errors, point source file format issues, and array subscript errors from oversized operation tables.

Recurring examples
  • forrtl: severe (408) — array subscript IDOP exceeds upper bound 1001 (tillage op)
  • forrtl: error (73): floating divided by zero — soil usersol table format
  • forrtl: severe (24): end-of-file during read — point source .dat file
  • SWAT run failed — output file locked by SWATCheck or SWATPlot
  • swat2009.exe not recognized — PATH not set or wrong executable name
  • Debug version vs release: which to use for diagnosing crashes
Systematic Solution

Pre-run input validator checking: usersol table column format, management operation counts against array bounds, point source file structure, output file lock status. Clear executable selection guide (debug vs release vs 32-bit vs 64-bit).

CAT-04 · Research Priority
Calibration & SWAT-CUP
TheoreticalProcedural
620
threads

SWAT-CUP calibration workflow, SUFI2 parameter selection, observation file formatting, multi-variable/multi-site calibration sequencing, and the disconnect between SWAT-CUP results and SWAT standalone runs when copying calibrated parameters back.

Recurring examples
  • Sediment output different in ArcSWAT vs SWAT-CUP after copying parameters
  • SWAT-CUP observation file column-width format (5 / 5 / 14.3)
  • Sensitivity analysis: all parameters show p-value = 0 — format error
  • Multi-variable calibration: hydrology first, then sediment, then nutrients
  • SWAT-CUP parallel processing: 200 simulations — some crash partway
  • How to prepare inputs for SWAT-CUP — all inputs successfully prepared but crashes
Systematic Solution

Document the exact parameter copy-back procedure from SWAT-CUP to ArcSWAT TxtInOut. Publish the correct observation file column-width format with a template. Standardise the multi-variable calibration sequence: streamflow → sediment → nutrients (documented but not universally followed).

CAT-05 · Scientific Core
Output Interpretation & Water Balance
TheoreticalInterpretation
500
threads

Understanding SWAT's output files (output.rch, output.sub, output.hru, output.std), extracting baseflow components, interpreting sediment yield = 0 scenarios, water balance closure, and the meaning of aggregation periods (daily/monthly/annual). The most theoretically rich category in SWAT-user.

Recurring examples
  • Sediment yield = 0 despite sediment-prone area — which MUSLE parameter is zero?
  • Baseflow trend incorrect — GW_DELAY, ALPHA_BF, RCHRG_DP interaction
  • How to extract baseflow components from output.sub (WYLD vs GW_Q)
  • Adding two hydrological stations at different points — interpretation
  • SWAT models overestimated results even after SWAT+ editor calibration
  • Values in basin_wb_aa not matching expected averages of annual outputs
Systematic Solution

Publish an Output Variables Guide: map every output column to its physical meaning. Include a baseflow separation worked example. Provide a water balance closure verification checklist. Document the aggregation rules for daily/monthly/annual output files.

CAT-06 · Scientific Gap
Sediment, Nutrients & Water Quality
TheoreticalProcedural
390
threads

SWAT's most scientifically substantive domain: sediment routing (MUSLE), nitrogen and phosphorus cycling, in-stream water quality, BMP evaluation, and crop parameter sensitivity. The proportion of theoretical issues in this category is higher than the model average, reflecting genuine scientific complexity.

Recurring examples
  • Sediment yield zero — which MUSLE factor (C, P, LS, K) is causing it?
  • NO3 loading expressed as NaN — soil nitrogen cycling error in NMINRL
  • BMP evaluation: how to implement filter strips, cover crops, reduced tillage
  • Crop parameter calibration for non-standard crops (olives, local varieties)
  • NLCD 2024 dataset upload into SWAT/HAWQS — new land use categories
  • Evaluating BMPs to reduce sediment yield under climate change scenarios
Systematic Solution

Publish a MUSLE diagnostic guide: which parameters control each component of the erosion equation. Document the standard BMP implementation procedures. Provide a crop parameter database for non-US crops. This category is a strong PhD publication topic for water quality applications.

🖥️
CAT-07 · Software Issues
ArcSWAT / QSWAT GUI & Database Errors
BugProcedural
275
threads

ArcSWAT-specific database (SWAT2012.mdb) corruption, version update issues requiring "Database Update" utility, ArcGIS version compatibility (10.x vs Pro), QSWAT interface bugs, and the "write input tables" workflow errors. ArcSWAT's dependency on Microsoft Access adds unique fragility.

Recurring examples
  • "SWAT2012.mdb database does not exist" — project database path broken
  • PR_W1_ vs PR_W1 field name bug in WGEN table — persists across versions
  • Database Update utility needed when upgrading ArcSWAT version
  • ArcSWAT not integrating with ArcGIS 10.8 — version compatibility
  • Output file locked — model ran but file opened in another program
  • SWAT2012 revision mismatch with SWAT-CUP executable
Systematic Solution

Maintain a versioned compatibility matrix (ArcGIS × ArcSWAT × SWAT2012 revision). Automate the Database Update step on project open. Deprecate the Access MDB dependency in future ArcSWAT releases — migrate to SQLite (as SWAT+ did). Publish a clear version upgrade checklist.

CAT-08 · Management & Advanced
Land Management, Reservoirs & BMPs
TheoreticalConfig
155
threads

Reservoir parameter setup (res table: volume-area, target storage), irrigation from reservoir vs shallow aquifer, dam/reservoir routing, and the difference between project database res table vs SWAT2012.mdb. Less frequent but scientifically significant for water management applications.

Recurring examples
  • Dam/reservoir setup: res table in project .mdb vs SWAT2012.mdb distinction
  • Irrigation source: reservoir defined but model drawing from shallow aquifer
  • Monthly target storage and consumptive use — when and how to populate
  • How to model multiple dams in series along main channel
Systematic Solution

Reservoir setup worked example covering all res table fields. Document the irrigation source hierarchy (reservoir vs aquifer vs main channel). Provide a multi-reservoir routing tutorial with example from a managed watershed.
